## ACT

### 1. Patch `compute_subscriber_state` (+~10 lignes)
- Nouveaux champs entries : `referrer_token` (None par défaut), `referrals` (int).
- Dict `referred_by[child_token] = parent_token` pendant le replay subscribe.
- Post-replay : pour chaque `(child, parent)`, si `state[child].status == "confirmed"` → `state[parent].referrals += 1`.
- Sécurité : `parent in state` vérifié (referrer_token validé en amont mais on garde la double check).

### 2. Patch POST `/api/subscribe` (+~12 lignes)
- Parse `referrer_token` (regex `^[A-Za-z0-9_-]+$` + len ≤ 96, sinon string vide).
- Après calcul de `state` pour idempotence, validation `referrer_token` :
  - doit exister dans `state`
  - `state[referrer_token].status == "confirmed"`
  - `state[referrer_token].email != email` (anti-self-referral cross-email)
- Si invalide → `valid_referrer = ""` mais signup continue normalement.
- Event `subscribe` enregistré avec `referrer_token: valid_referrer or None`.

### 3. Nouvel endpoint GET `/api/me` (+~22 lignes)
- Validation token (regex + len ≤ 96).
- Replay state, retourne `{ok: true, email_masked, topic, status, referral_url, referrals}`.
- `email_masked` : `"al***@domain.tld"` (premier 2 chars + ***).
- 404 si token inexistant, 400 si malformé.

### 4. Bloc `_referral_block_html` (+~22 lignes helper)
- HTML inline style (Tailwind utility-like cohérent SUBSCRIBE_PAGE_CSS) : panel `#f1f5f9` + bordure `#cbd5e1`.
- Texte "Parrainez d'autres bailleurs ou locataires" + compteur en bold.
- Input readonly avec ref_url + 3 boutons : WhatsApp (`wa.me/?text=`), Email (`mailto:`), Copy (button + IIFE 1 ligne `navigator.clipboard.writeText`).
- Helper `_url_encode` (`urllib.parse.quote(s, safe="")`) pour les liens partage.
- Injecté dans `/api/confirm` cas `confirmed` (déjà confirmé) + cas fresh-confirm (`fresh_state` recompute pour ref_count correct post-event).

### 5. Patch `/api/stats` (+~3 lignes)
